# 평가: 데이터셋, 실험, LLM-as-a-Judge

> 키워드: 데이터셋, Dataset, 실험, experiment, 평가, LLM-as-a-Judge, 스코어, score, 골든 답변, 회귀 테스트

## 감(感)이 아니라 데이터로 개선하기

"프롬프트를 고쳤더니 좋아진 것 같다"는 개선이 아니라 추측이다.
개선 사이클은 **실패 사례 수집 → 데이터셋화 → 변경 → 실험 → 점수 비교**로 닫혀야 한다.
이것이 루프 엔지니어링의 두 번째 층위, 개발 루프다.

## 실패 트레이스를 데이터셋으로

운영 중 낮은 점수(사용자 피드백, 자동 평가)가 붙은 트레이스는 가장 값진 자산이다.
이런 hard case들을 골라 **데이터셋 항목**(input: 질문, expected_output: 골든 답변)으로
승격시키면, 이후 모든 변경을 같은 시험지로 채점할 수 있는 회귀 테스트가 된다.

## 실험 (Experiment / Run)

데이터셋의 전 항목에 대해 "변경 전 버전"과 "변경 후 버전"을 각각 실행한 것이 실험 run이다.
예: 프롬프트 v1으로 한 번, v2로 한 번 돌려 run 두 개를 만들고 나란히 비교한다.
프롬프트를 코드에 하드코딩하지 않고 프롬프트 관리 도구에서 버전으로 관리하면
실험과 롤백이 훨씬 쉬워진다.

## LLM-as-a-Judge

수백 개 결과를 사람이 일일이 채점할 수 없으므로, 채점 자체를 LLM에게 맡긴다.
judge 프롬프트에 (질문, 골든 답변, 실제 답변)을 주고 충실도를 0~1로 매기게 하는 방식이 기본이다.
judge는 완벽하지 않지만 **일관적**이므로 버전 간 상대 비교에는 충분히 유효하다.
스코어는 트레이스나 실험 항목에 부착되어 대시보드에서 집계된다.

## 안티패턴

- 평가 없이 프롬프트를 계속 덮어쓰기 (무엇이 좋아졌는지 알 수 없음)
- 성공 사례로만 데이터셋 구성 (회귀를 못 잡음)
- judge 프롬프트를 검증 없이 신뢰 (judge도 평가 대상이다)
