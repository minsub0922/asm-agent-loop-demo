# Agentic RAG 패턴

> 키워드: RAG, 검색 증강 생성, 관련성 평가, grade_documents, 쿼리 재작성, transform_query, Adaptive RAG, 라우팅, 벡터스토어

## Naive RAG의 한계

기본 RAG는 `검색 → 생성` 두 단계 직선 파이프라인이다.
문제는 검색이 빗나갔을 때다. 관련 없는 문서가 검색되어도 파이프라인은 멈추지 않고,
LLM은 엉뚱한 문맥을 근거로 그럴듯한 오답을 만들어낸다.
사용자의 어휘와 문서의 어휘가 다를 때(영어 섞인 구어체 질문 등) 특히 자주 실패한다.

## 문서 관련성 평가 (grade_documents)

검색 직후 "이 문서가 질문에 답하는 데 쓸모 있는가?"를 LLM에게 이진 판정(yes/no)시키는 노드를 넣는다.
관례적으로 `grade_documents`라 부른다. 판정은 저렴한 모델로 충분하며,
쓸모없는 문서를 걸러내는 것만으로 생성 품질이 크게 오른다.

## 쿼리 재작성 (transform_query)

관련 문서가 하나도 없으면 그대로 생성하는 대신, **질문을 검색 친화적으로 재작성**해서
검색 노드로 되돌아간다. 관례적으로 `transform_query`라 부른다.
구어체·외국어 표현을 문서의 표준 용어로 바꾸는 것이 핵심이다.
이 되돌아가는 조건부 엣지가 그래프에 **루프**를 만들며, 여기서부터 시스템은 자기 수정 능력을 가진다.
루프에는 반드시 최대 재작성 횟수 같은 **탈출 조건**을 상태에 카운터로 두어야 한다.

## 질문 라우팅 (Adaptive RAG)

모든 질문이 벡터스토어로 갈 필요는 없다. 진입점에 **라우터**를 두고 질문 성격에 따라 분기한다.

- `vectorstore`: 내부 문서(코퍼스)로 답할 수 있는 질문
- `web_search`: 최신 정보·외부 지식이 필요한 질문
- `direct_answer`: 검색 자체가 불필요한 인사말·잡담·단순 계산

라우터 + 관련성 평가 + 쿼리 재작성을 모두 갖춘 형태를 Adaptive RAG라 부른다.
질문마다 실행 경로가 달라지므로, 관측 도구로 경로를 추적하는 일이 중요해진다.
